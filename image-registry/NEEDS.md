# image-registry — چه چیزی نیاز داریم (به‌روزشده ۲۰۲۶-۰۷-۰۵)

## وضعیت فعلی
سرویس تازه (ساخته‌شده ۲۰۲۶-۰۷-۰۴) — جایگزین whitelist محلیِ image در `agentmanager`. از همان روز،
علاوه بر whitelist، خودِ فایل image (خروجی `docker save`) را هم می‌تواند ذخیره و توزیع کند — **ولی این
قسمت فقط سمتِ خودِ image-registry پیاده شده**؛ رجوع بخش ۰ زیر، این مهم‌ترین یافته‌ی بررسی امروز است.

## ۰. یافته‌های بحرانیِ بررسی مجدد ۲۰۲۶-۰۷-۰۵ (این‌ها را اول بخوانید)

1. **agentmanager هنوز از قابلیت دانلود+load استفاده نمی‌کند.** کد فعلی `image-registry` کامل و درست
   است (آپلود، اعتبارسنجی tar، sha256، دانلود با Range support) — ولی
   `agentmanager/internal/registryclient/registryclient.go`'s `IsAllowed` فقط فیلد `allowed` پاسخ
   `/v1/check` را می‌خواند، به `download_url`/`sha256` که همان پاسخ برمی‌گرداند اصلاً نگاه نمی‌کند.
   `agentmanager/internal/docker/client.go`'s `Deploy()` هنوز دقیقاً رفتار قدیمی را دارد: انتظار دارد
   image از قبل دستی روی سرور `docker load` شده باشد. یعنی **قابلیت اصلیِ «ثبت یک‌بار، توزیع خودکار به
   همه‌ی سرورها» که هدف این بخش از کار بود، هنوز عملاً فعال نیست** — فقط whitelist (بدون فایل) واقعاً
   استفاده می‌شود. جزئیات کامل implementation لازم: `agentmanager/NEEDS.md`.
2. **apimanager یک هدر اشتباه می‌فرستد و به احتمال زیاد ۴۰۳ می‌گیرد.**
   `apimanager/internal/handler/image_registry.go` هنگام proxy کردن `/v1/images*` یک `X-Admin-Key`
   می‌فرستد — ولی این مسیرها اصلاً به آن هدر نگاه نمی‌کنند (فقط `/v1/callers/*` این‌طور است). تصمیم
   واقعی `/v1/images*` صرفاً بر اساس IP خروجیِ خودِ apimanager + فیلد `CanWrite` روی آن IP در جدول
   `AllowedCaller` است. تا وقتی IP واقعیِ apimanager (نه یک کلید) به‌عنوان یک `AllowedCaller` با
   `CanWrite=true` ثبت نشود، هر تلاش ثبت/آپلود/حذف image از پنل وب `403` می‌گیرد. جزئیات و راه‌حل‌های
   پیشنهادی: `apimanager/NEEDS.md`.

## چیزی که واقعاً کم است
1. **بدون تست unit** — خصوصاً `internal/ipallow`'s `Check` (منطق CIDR/domain matching) که دقیقاً همان
   چیزی است که اگر اشتباه کار کند، یا همه‌چیز را رد می‌کند (fail-closed بی‌ضرر) یا بدتر — کسی را که
   نباید، اجازه می‌دهد.
2. **بدون Kubernetes manifest** و **بدون جا در `deploy/Makefile`'s `build-all`** — چون همین امروز
   ساخته شد، طبیعتاً هنوز جایی اضافه نشده؛ باید هم‌زمان با هر رفع gap مشابه در سایر سرویس‌ها
   (رجوع `agentmanager/NEEDS.md`) انجام شود.
3. **بدون رابط مدیریتی (وب یا تلگرام)** برای ثبت image/caller — همه‌چیز فعلاً فقط با `curl`/API خام
   قابل انجام است. با توجه به اینکه `apimanager/web` حالا یک پنل ادمین کامل دارد (رجوع
   `apimanager/NEEDS.md`)، یک صفحه‌ی ساده «مدیریت image-registry» (لیست image ها، لیست callerها، یک
   دکمه‌ی افزودن) طبیعی‌ترین جای بعدی برای این است.
4. **پشت یک reverse proxy واقعاً کار نمی‌کند** بدون تنظیم اضافه (رجوع README، بخش «محدودیت‌های
   شناخته‌شده») — قبل از استقرار پشت هر load balancer، این باید حل شود.
5. **`ADMIN_KEY` نمونه در `.env` باید rotate شود** قبل از هر استقرار واقعی.
6. **هیچ audit log ای برای تغییرات caller/image ثبت نمی‌شود** — یعنی اگر یک `CanWrite=true` اشتباه یا
   یک image نادرست ثبت شود، هیچ تاریخچه‌ای برای بررسی بعدی نیست. با توجه به اینکه این سرویس مستقیماً
   کنترل می‌کند «چه کدی روی سرورهای پلتفرم اجرا می‌شود»، این یک نیاز امنیتی جدی (نه فقط راحتی) است.
~~7. بدون سقف اندازه‌ی آپلود~~ — حل شد (۲۰۲۶-۰۷-۰۴): `MAX_IMAGE_FILE_SIZE_MB` (پیش‌فرض ۸۱۹۲ مگابایت)
   قبل از نوشتن روی دیسک چک می‌شود.
~~8. بدون اعتبارسنجی محتوای فایل آپلودشده~~ — حل شد (۲۰۲۶-۰۷-۰۴): `validateDockerSaveTar` در
   `internal/api` مطمئن می‌شود آرشیو واقعاً `docker save` است و شامل همان `name:tag`. توجه: این فقط
   ساختار tar/manifest را چک می‌کند، نه امنیت/سلامت خودِ لایه‌های image (آن یک لایه‌ی دیگر است — رجوع
   README «محدودیت‌های شناخته‌شده»).
9. **بدون پشتیبان‌گیری (backup) از `IMAGE_STORAGE_DIR`** — (رفع جزئی ۲۰۲۶-۰۷-۰۵) `docker-compose.yml`
   حالا یک named volume (`image_registry_data`) روی این مسیر mount می‌کند، پس دیگر با یک
   `docker-compose up --force-recreate` ساده گم نمی‌شود؛ ولی این هنوز backup واقعی نیست — از دست رفتن
   خودِ volume/دیسک هاست همچنان یعنی از دست رفتن همه‌ی آرتیفکت‌ها. نیاز به یک استراتژی backup منظم (یا
   مهاجرت به یک object storage مثل S3/MinIO) هنوز باقی است.
10. **بدون آپلود resumable/chunked** — قطع شدن یک آپلود بزرگ یعنی باید از اول تکرار شود.
11. **بدون rate limit روی دانلود** — یک caller (حتی مجاز) می‌تواند بارها فایل‌های بزرگ را دانلود کند و
    پهنای‌باند/IO سرویس را اشغال کند؛ فعلاً هیچ محدودیتی روی این نیست.

## این‌ها را در سرویس‌های دیگر هم نوشتم
- `agentmanager/NEEDS.md` — طرف مصرف‌کننده‌ی همین سرویس (پیاده‌سازی واقعی دانلود+load).
- `apimanager/NEEDS.md` — مشکل هدر/IP در proxy پنل ادمین.

## کلاینت سمت مصرف‌کننده (shared) — هنوز merge نشده
یک پیاده‌سازی مرجع کامل برای مصرف این سرویس (`Check` + `EnsureLoaded`: دانلود فایل، verify
sha256، `docker load`) نوشته و تحویل داده شد، با این مسیر مقصد پیشنهادی:
`shared/pkg/adapters/imageregistry/client.go` — دقیقاً هم‌الگو با
`shared/pkg/adapters/nats` که همین سرویس هم استفاده می‌کند. چرا داخل `shared` نه داخل خودِ
`agentmanager`: هر سرویس دیگری که در آینده بخواهد deploy کند هم باید بتواند همین کلاینت را
ایمپورت کند، نه اینکه دوباره از صفر بنویسدش.

**این فایل هنوز داخل ریپوی واقعی `shared` قرار نگرفته** — چون آن ریپو در دسترس مستقیم نبود، فقط
تحویل داده شد. قبل از استفاده‌ی واقعی باید: (۱) در مسیر بالا کپی شود، (۲) امضای دقیق
`shared/pkg/ports.Logger`/`ports.F` با آنچه در فایل فرض شده تطبیق داده شود، (۳) در
`agentmanager/internal/docker/client.go` جایگزین منطق قدیمیِ `docker pull`/`isImageAllowed` شود،
(۴) `agentmanager` باید `IMAGE_REGISTRY_URL` را به `imageregistry.Config.BaseURL` بدهد.

## به‌روزرسانی ۲۰۲۶-۰۷-۰۶: جداسازی دیتابیس
`image-registry` حالا دیتابیس مخصوص خودش (`imageregistry`) را دارد — دیگر با بقیه‌ی سرویس‌های
مرکزی مشترک نیست (رجوع `deploy/migrations/000_create_databases.sql`).
