# image-registry

سرویس جدید (ساخته‌شده ۲۰۲۶-۰۷-۰۴) — یک **استخر ثبت‌شده‌ی image:tag های مجاز**، به‌همراه کنترل دسترسی
بر اساس **IP/CIDR واقعی فراخوان** (نه یک کلید API به‌تنهایی). این سرویس جایگزین whitelist محلیِ قدیمی
`agentmanager` شد (env var `ALLOWED_IMAGES`، prefix matching داخل خودِ کد هر سرور).

## این سرویس دقیقاً چیست

قبل از این، هر `agentmanager` (روی هر سرور) یک لیست prefix محلی در `.env` خودش داشت
(`ALLOWED_IMAGES=creatorbot/,registry.local/creatorbot/`) و خودش تصمیم می‌گرفت یک image مجاز است یا نه.
مشکلات این روش: (۱) هیچ «ثبت» واقعی وجود نداشت — هرچیزی که prefix درست داشت مجاز بود، نه یک لیست دقیق
از image:tag های واقعاً تأییدشده؛ (۲) هیچ کنترلی روی *چه کسی* اجازه‌ی deploy کردن دارد نبود — فقط
*چه چیزی* deploy می‌شود.

`image-registry` این دو مشکل را حل می‌کند:

1. **استخر ثبت** (`RegisteredImage`): هر image:tag که واقعاً برای deploy تأیید شده، دقیقاً با نام کامل
   ثبت می‌شود (نه یک prefix حدسی) — به‌همراه نوع سرویس و توضیح، برای خوانایی.
2. **whitelist بر اساس IP/دامنه** (`AllowedCaller`): هر سروری که اجازه دارد از این سرویس بپرسد یا در
   آن ثبت کند، با IP/CIDR خودش (و اختیاراً یک دامنه) در این سرویس ثبت می‌شود. یک caller می‌تواند
   **فقط بخواند** (`GET /v1/check`, `GET /v1/images`) یا **بنویسد هم بکند** (ثبت/حذف image، یا حتی
   مدیریت callerهای دیگر) — این دقیقاً همان چیزی است که خواسته شده بود: «اگه از اون IP بود، فقط بتواند
   فلان کار را بکند».

سرویس **هیچ کار دیگری نمی‌کند** — نه پول، نه ربات، نه هیچ منطق کسب‌وکار دیگری. فقط نگه‌داری این دو
جدول و آرتیفکت‌های واقعی image (رجوع بخش بعدی).

## ذخیره‌سازی و توزیع خودِ فایل image (نه فقط whitelist)

نسخه‌ی اول این سرویس فقط یک whitelist از نام‌ها بود — تصمیم می‌گرفت یک `name:tag` مجاز است یا نه، ولی
خودِ فایل image را از یک registry بیرونی (Docker Hub یا مشابه) با `docker pull` می‌گرفت. این کافی
نبود: اگر روی یک سرور تازه `agentmanager` بالا می‌آمد و آن image را نداشت، وابسته به دسترسی آن سرور به
یک registry بیرونی بود. حالا `image-registry` خودش **منبع واقعی فایل** هم هست:

1. بعد از `POST /v1/images` (ثبت متادیتا)، فایل واقعی با `POST /v1/images/:id/file` آپلود می‌شود —
   یک multipart با فیلد `file`، دقیقاً خروجی `docker save name:tag -o file.tar`.
2. سرویس فایل را زیر `IMAGE_STORAGE_DIR` (پیش‌فرض `/data/images`) ذخیره می‌کند، محتوایش را چک
   می‌کند (باید واقعاً یک `docker save` معتبر باشد و شامل همان `name:tag`، وگرنه رد و پاک می‌شود)،
   چک‌سام `sha256` می‌گیرد و اندازه‌اش را ثبت می‌کند. اگر حجم از `MAX_IMAGE_FILE_SIZE_MB` (پیش‌فرض
   ۸ گیگابایت) بیشتر باشد، حتی قبل از نوشتن روی دیسک رد می‌شود.
3. `GET /v1/check` حالا اگر فایل موجود باشد، `has_file: true` به‌همراه `download_url`، `sha256` و
   `size` برمی‌گرداند.
4. `agentmanager` به‌جای `docker pull` از یک registry بیرونی، همان `download_url` را (نسبت به
   `IMAGE_REGISTRY_URL` خودش) با `GET /v1/images/:id/download` می‌گیرد، چک‌سام دریافتی را با
   `sha256` برگشتی مقایسه می‌کند، و بعد محلی `docker load -i file.tar` می‌زند.

این یعنی deploy روی هر سرور جدید فقط به دو چیز نیاز دارد: دسترسی HTTP به `image-registry` (که به‌هرحال
لازم بود) و **هیچ دسترسی دیگری به هیچ registry بیرونی**. مسیر دانلود هم پشت همان `ipGate` است — یعنی
فقط callerهای ثبت‌شده (IP-based) می‌توانند فایل را بگیرند، دقیقاً همان مدل امنیتی بقیه‌ی سرویس.

**نکته‌ی عملیاتی مهم**: `IMAGE_STORAGE_DIR` باید روی یک volume پایدار mount شود (رجوع `Dockerfile`)،
وگرنه با هر redeploy کانتینر سرویس، همه‌ی فایل‌های آپلودشده گم می‌شوند (فقط ردیف‌های DB با
`file_path` اشاره به فایلی که دیگر وجود ندارد باقی می‌مانند).

## چرا این سرویس با HTTP کار می‌کند، نه NATS (تنها استثنای این پلتفرم)

قانون بنیادی کل پروژه این است که همه‌ی ارتباط بین‌سرویسی روی NATS باشد. این سرویس عمداً این قانون را
نمی‌شکند، بلکه یک محدودیت واقعی را دور می‌زند: **NATS یک message bus است — پیام‌رسانی‌اش IP واقعیِ
فرستنده را در سطح اپلیکیشن در اختیار responder نمی‌گذارد**، فقط یک ادعای داخل payload که به‌راحتی
جعل‌پذیر است (دقیقاً همان درسی که در ممیزی امنیتی ۲۰۲۶-۰۷-۰۲ از باگ `ServiceKey` در `botpay` گرفته شد:
هیچ‌وقت فقط به یک مقدار داخل payload به‌عنوان راز اعتماد نکن). یک اتصال HTTP مستقیم، برعکس، IP واقعیِ
TCP را در سطح شبکه به سرور می‌دهد — این تنها راهی است که «whitelist بر اساس IP» واقعاً به‌معنای امنیتی
درست (نه فقط یک ادعا) اجرا شود. به همین دلیل، `image-registry` تنها سرویس این پلتفرم است که با HTTP
ساده کار می‌کند.

## مدل داده

- `RegisteredImage`: `Name` (بدون تگ، مثل `creatorbot/uploader-bot`)، `Tag`، `ServiceType` (اختیاری،
  برای خوانایی)، `Description`، `IsActive`. یکتا روی `(Name, Tag)`.
- `AllowedCaller`: `Label` (نام خوانا)، `CIDR` (اجباری — `net.ParseCIDR`؛ یک IP تکی هم به‌صورت
  `1.2.3.4/32` قابل‌قبول است)، `Domain` (اختیاری، لایه‌ی دوم — رجوع به «محدودیت‌های شناخته‌شده»)،
  `CanWrite` (فقط این caller ها می‌توانند ثبت/حذف کنند)، `IsActive`.

## API

همه‌ی مسیرهای `/v1/*` (به‌جز مدیریت callerها) پشت `ipGate` هستند — یعنی اول IP فراخوان با جدول
`AllowedCaller` مطابقت داده می‌شود؛ بدون تطابق → `403`.

| Method | مسیر | نیاز به CanWrite | کاربرد |
|---|---|---|---|
| GET | `/health` | خیر (بدون IP gate) | health check |
| GET | `/v1/check?name=&tag=` | خیر | agentmanager قبل از هر deploy این را صدا می‌زند؛ اگر فایل موجود باشد `download_url`/`sha256`/`size` هم برمی‌گرداند |
| GET | `/v1/images` | خیر | لیست همه‌ی image های ثبت‌شده |
| GET | `/v1/images/:id/download` | خیر | دانلود فایل واقعی image (خروجی `docker save`) |
| POST | `/v1/images` | **بله** | ثبت متادیتای image تازه |
| POST | `/v1/images/:id/file` | **بله** | آپلود/رونویسیِ فایل واقعی image (multipart، فیلد `file`) |
| PATCH | `/v1/images/:id` | **بله** | فعال/غیرفعال کردن |
| DELETE | `/v1/images/:id` | **بله** | حذف (فایل روی دیسک هم پاک می‌شود) |

مدیریت `AllowedCaller` (`GET/POST/PATCH/DELETE /v1/callers`) پشت `ipGate` نیست، پشت یک هدر
`X-Admin-Key` ثابت از env (`ADMIN_KEY`) است — چون قبل از ثبت اولین caller، هیچ IP ای اصلاً `CanWrite`
ندارد (مشکل مرغ‌وتخم‌مرغ). این تنها راه bootstrap است.

### Bootstrap (اولین راه‌اندازی)

دو راه، هرکدام کافی است:
1. `SEED_CALLER_CIDR`/`SEED_CALLER_LABEL` را در `.env` این سرویس پر کنید (IP/CIDR ماشین یا سرور
   ادمین) — در اولین اجرا، اگر جدول callerها خالی باشد، یک caller با `CanWrite=true` خودکار ساخته
   می‌شود.
2. یا مستقیم با `X-Admin-Key: <ADMIN_KEY>` روی `POST /v1/callers` یک caller بسازید.

### ⚠️ نکته‌ی مهم برای هر سرویس دیگری که این API را proxy می‌کند (مثل apimanager)

`X-Admin-Key` **فقط** روی مسیرهای `/v1/callers/*` چک می‌شود. مسیرهای `/v1/images*` (ثبت/آپلود/ویرایش/
حذف image) اصلاً به این هدر نگاه نمی‌کنند — آن‌ها فقط پشت `ipGate`+`requireWrite` هستند، یعنی تصمیم
صرفاً بر اساس IP واقعیِ فراخوان + فیلد `CanWrite` روی آن IP در جدول `AllowedCaller` گرفته می‌شود.

این دقیقاً همان چیزی است که در بررسی ۲۰۲۶-۰۷-۰۵ در `apimanager/internal/handler/image_registry.go`
پیدا شد: آن‌جا یک `X-Admin-Key` (از env `IMAGE_REGISTRY_ADMIN_KEY`) روی همه‌ی درخواست‌ها (شامل
`/v1/images*`) فرستاده می‌شود — که برای مسیرهای image بی‌اثر است. اگر IP خروجیِ خودِ `apimanager`
به‌عنوان یک `AllowedCaller` با `CanWrite=true` در این سرویس ثبت نشده باشد، هر تلاش برای ثبت/آپلود/حذف
image از پنل وب با `403` شکست می‌خورد — حتی با اینکه `X-Admin-Key` درست فرستاده شده. رجوع
`apimanager/NEEDS.md` برای جزئیات کامل و راه‌حل.

## agentmanager چطور از این استفاده می‌کند

### وضعیت واقعیِ امروز (تأییدشده با خواندن مستقیم کد در بررسی ۲۰۲۶-۰۷-۰۵)

⚠️ **فقط نیمه‌ی «چک» پیاده شده، نیمه‌ی «دانلود+load» هنوز نه.** `agentmanager/internal/registryclient`
فقط `IsAllowed(ctx, name, tag)` دارد که `GET /v1/check` را صدا می‌زند و صرفاً `allowed: true/false` را
می‌خواند — به `download_url`/`sha256` که همین endpoint برمی‌گرداند اصلاً نگاه نمی‌کند.
`agentmanager/internal/docker/client.go`'s `Deploy()` هنوز دقیقاً همان رفتار قدیمی را دارد: بعد از چک
whitelist، مستقیم `ImageInspectWithRaw` می‌زند و اگر image محلی نبود، خطای *«ابتدا image را روی سرور
بسازید/لود کنید»* برمی‌گرداند — یعنی هنوز کسی باید دستی روی هر سرور `docker load` بزند. **بخش‌های
پایین همین بخش، طراحیِ مقصود/هدف را توضیح می‌دهند، نه چیزی که همین امروز اجرا می‌شود** — قبل از این
بررسی، این تمایز در این فایل مشخص نبود و می‌توانست گمراه‌کننده باشد. رجوع `agentmanager/NEEDS.md` برای
کاری که باید انجام شود تا این طراحی واقعاً پیاده شود.

### طراحی مقصود (هنوز پیاده نشده)

`agentmanager` باید به‌جای `docker pull`، این‌طور کار کند:
1. `GET {IMAGE_REGISTRY_URL}/v1/check` — اگر پاسخ `has_file: true` باشد، `download_url`/`sha256`/`size`
   هم همراهش می‌آید.
2. `GET {IMAGE_REGISTRY_URL}{download_url}` — فایل tar را می‌گیرد (پشت همان `ipGate`).
3. `sha256` فایل دریافتی را با مقدار برگشتی مقایسه کند — عدم تطابق یعنی رد deploy (فایل خراب/ناقص،
   fail-closed).
4. `docker load -i file.tar` را محلی روی همان سرور اجرا کند — دیگر نیازی به `docker pull` و در نتیجه
   نیازی به دسترسی آن سرور به هیچ registry بیرونی (خصوصی یا عمومی) نباشد.

اگر `has_file: false` باشد (فقط متادیتا ثبت شده، هنوز فایلی آپلود نشده)، deploy آن image باید رد شود
(fail-closed) تا وقتی فایل واقعی هم آپلود شود.

برای اینکه یک `agentmanager` واقعاً بتواند deploy کند (چه امروز، چه بعد از پیاده‌شدن طراحی بالا)، باید:
1. IP/CIDR سروری که آن `agentmanager` رویش اجرا می‌شود، در `image-registry` به‌عنوان یک
   `AllowedCaller` (حداقل read-only) ثبت شده باشد.
2. image:tag ای که `botmanager`/ادمین می‌خواهد deploy کند، از قبل در `RegisteredImage` ثبت و فعال باشد.

## نکته‌ی عملیاتی مهم: docker-compose محلی در برابر سرورهای واقعی

طبق معماری این پلتفرم، `agentmanager` واقعی روی سرورهای فیزیکی جداگانه اجرا می‌شود (نه داخل همان
`docker-compose.yml` مرکزی) و از راه اینترنت با IP عمومی واقعی خودش به سرویس‌های مرکزی وصل می‌شود.
اگر برای توسعه/تست محلی هر دو سرویس را در همان یک `docker-compose.yml` بالا بیاورید، IP ای که
`image-registry` می‌بیند، IP داخلیِ شبکه‌ی Docker (مثل `172.x.x.x`) خواهد بود، نه IP واقعیِ یک سرور —
برای این حالت، `SEED_CALLER_CIDR` را روی بازه‌ی شبکه‌ی داخلی Docker (مثلاً `172.16.0.0/12`) تنظیم کنید.
برای production واقعی، `AllowedCaller.CIDR` باید IP عمومیِ واقعیِ هر سرور agentmanager باشد.

## محدودیت‌های شناخته‌شده (صادقانه)

- **پشت یک reverse proxy/load balancer کار نمی‌کند بدون تنظیم اضافه**: این سرویس IP را مستقیم از
  `RemoteAddr` اتصال TCP می‌خواند، نه از هدرهایی مثل `X-Forwarded-For` (که به‌راحتی توسط کلاینت قابل
  جعل‌اند اگر پشت یک proxy مورد اعتماد فیلتر نشوند). اگر بین agentmanager و image-registry یک
  load balancer/reverse proxy بگذارید، باید یا مستقیماً IP واقعی را عبور دهد (مثل `proxy_protocol` در
  nginx)، یا این سرویس را طوری تغییر دهید که فقط از پشت همان proxy مشخص به `X-Forwarded-For` اعتماد
  کند.
- **بررسی Domain یک لایه‌ی دوم است، نه منبع اصلی اعتماد**: چون DNS می‌تواند کش شود، دیر به‌روز شود، یا
  (در تئوری) مسموم شود، `Domain` روی یک `AllowedCaller` فقط علاوه بر `CIDR` چک می‌شود، هرگز جایگزینش
  نمی‌شود.
- **بدون تست unit، بدون Kubernetes manifest** — رجوع `NEEDS.md` در همین پوشه.
- **agentmanager هایی که پشت NAT مشترک هستند** (چند سرور با یک IP خروجی عمومی) نمی‌توانند با CIDR
  دقیق از هم تفکیک شوند — باید یا NAT جداگانه داشته باشند، یا از یک CIDR بازه‌ای مشترک برای همه‌ی آن
  گروه استفاده کنید (که یعنی نمی‌شود بین آن‌ها تفکیک دسترسی نوشتن کرد).
- **سقف اندازه‌ی آپلود و اعتبارسنجی محتوا** حالا اعمال می‌شود: `MAX_IMAGE_FILE_SIZE_MB` (پیش‌فرض
  ۸۱۹۲ مگابایت) رد می‌کند قبل از نوشتن روی دیسک، و `validateDockerSaveTar` مطمئن می‌شود آرشیو واقعاً
  یک `docker save` است و شامل همان `name:tag` ثبت‌شده — نه هر فایل دلخواهی با پسوند `.tar`. این چک روی
  محتوای tar است (manifest.json/repositories)، نه یک ضمانت کامل امنیتی روی خودِ لایه‌های image.
- **بدون آپلود قابل‌ازسرگیری (resumable/chunked)**: اگر آپلود یک فایل چند گیگابایتی وسط راه قطع شود،
  باید کامل از اول آپلود شود.
- **فایل‌ها روی یک دیسک محلی**، نه یک object storage توزیع‌شده — یعنی این سرویس هنوز single point of
  failure برای *وجود فیزیکی* فایل‌هاست (رجوع `NEEDS.md` برای پشتیبان‌گیری).
