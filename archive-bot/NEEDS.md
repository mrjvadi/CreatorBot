# archive-bot — چه چیزی نیاز داریم (بررسی ۲۰۲۶-۰۷-۰۴)

## وضعیت فعلی (خلاصه‌ی واقعی، نه قدیمی)

بررسی تازه انجام شد. این کوچک‌ترین سرویس از ۴ سرویس بررسی‌شده است: فقط ۹
فایل Go، ~۹۴۱ خط. برخلاف uploader-bot/vpn-bot/member-bot، هیچ رشد پنهانی
اینجا دیده نشد — اندازه و پیچیدگی سرویس دقیقاً با ادعای CLAUDE.md («آرشیو
فایل با جستجوی فارسی fuzzy») همخوانی دارد؛ چیزی بزرگ‌تر یا کوچک‌تر از
انتظار نبود.

نکات مثبت:
- مدل ادمین بسیار ساده و امن است: `isAdmin` فقط یک owner ثابت را چک می‌کند
  (`internal/tgbot/handler.go:43`) — نه سیستم چندسطحی، ولی چون فقط یک نفر
  (owner) قرار است مدیریت کند، این با نیاز واقعی سرویس همخوانی دارد، نه یک
  نقص.
- همه‌ی مسیرهای ادمین (`onAdd`, `onNewCategory`, `onMedia`, `del` callback)
  واقعاً `isAdmin` را چک می‌کنند — نمونه‌ها:
  `internal/tgbot/search_handler.go:48-51`, `internal/tgbot/handler.go:116,
  141, 195`.
- جستجوی fuzzy فارسی (`internal/search/search.go`) از پارامترهای SQL درست
  استفاده می‌کند (`similarity(...) > 0.1`, `Where(..., ?)`) — بدون خطر
  SQL injection، حتی با ورودی آزاد کاربر.
- کامنت داخل خود کد (`internal/search/search.go:44-47`) توضیح می‌دهد این
  تابع طوری طراحی شده که بشود بعداً به Elasticsearch/Meilisearch سوییچ
  کرد بدون تغییر handler — طراحی تمیز.

## چیزی که واقعاً کم است (با file:line برای هر ادعا)

- **صفر فایل تست** — هیچ `_test.go` در کل سرویس. با توجه به سادگی نسبی
  سرویس، ریسک این کمتر از بقیه است، ولی حداقل تست واحد برای
  `search.Normalize` (`internal/search/search.go:20-40`، منطق نرمال‌سازی
  حروف عربی/فارسی) به‌خاطر حساسیت‌های یونیکد ارزش نوشتن دارد.
- هیچ TODO/FIXME یا مدل/handler یتیم پیدا نشد. `RelatedFiles`
  (`internal/search/search.go:68-84`) تعریف شده — بررسی شد که آیا استفاده
  می‌شود یا نه:

```
grep -rn "RelatedFiles" archive-bot --include=*.go
```
  فقط در همان فایل تعریف شده و در `internal/tgbot` هیچ‌جا صدا زده نمی‌شود —
  یعنی «فایل‌های مرتبط» یک قابلیت آماده ولی وصل‌نشده به UI است
  (`internal/search/search.go:68`, تابع صادراتی `RelatedFiles`، صفر فراخوان
  در `internal/tgbot/*.go`). این تنها یتیمِ واقعی این سرویس است — کوچک و
  کم‌ریسک، صرفاً یک قابلیت «پیشنهاد فایل مشابه» که آماده است ولی هیچ
  دکمه/handler آن را صدا نمی‌زند.

جمع‌بندی صادقانه: archive-bot از نظر کیفیت کد و پوشش امنیتی وضعیت خوبی
دارد؛ تنها کمبودها تست و یک قابلیت آماده‌ولی-وصل‌نشده (RelatedFiles) است.

## این‌ها را در سرویس‌های دیگر هم نوشتم (اگر مرتبط بود)

یادداشتی به `apimanager/NEEDS.md` اضافه شد: مدیریت دسته‌بندی‌ها
(`internal/models/models.go:29-33`, مدل `Category`) و لیست فایل‌های آرشیو
(`internal/models/models.go:37-47`, مدل `File`) کاندیدای ساده‌ای برای
CRUD وب هستند — چون این سرویس فقط تک‌ادمین است، پیچیدگی کمی برای اتصال به
apimanager دارد در مقایسه با uploader-bot/vpn-bot.
