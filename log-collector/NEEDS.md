# log-collector — چه چیزی نیاز داریم (بررسی ۲۰۲۶-۰۷-۰۴)

## وضعیت فعلی
سرویس تازه (ساخته‌شده ۲۰۲۶-۰۷-۰۲) — ۶ فایل، ۵۷۰ خط. جمع‌آوری لاگ Warn+ از همه‌ی سرویس‌ها روی
`logs.events`، ذخیره در Mongo، API کوئری، هشدار تلگرام با topic اختصاصی هر سرویس. جزئیات کامل:
`docs/services/log-collector.md`.

## چیزی که واقعاً کم است
1. **بدون صفحه‌ی وب** — با اینکه یک `GET /logs` کامل با فیلتر دارد، هیچ صفحه‌ای در `apimanager/web`
   آن را صدا نمی‌زند؛ کاربر باید یا مستقیم API را کوئری کند یا فقط از تلگرام (اگر تنظیم شده) استفاده
   کند. رجوع `apimanager/NEEDS.md`.
2. ✅ **رفع شد در همین بررسی (۲۰۲۶-۰۷-۰۴)**: `source-service` واقعاً یک کلاینت NATS کامل داشت (سیستم
   worker/rules خودش) ولی به `log.AttachNATS` وصل نبود — یک خط اضافه شد در `cmd/service/main.go`.
3. **بدون تست unit و بدون Kubernetes manifest.**
4. **مسیر end-to-end (logger→NATS→Mongo→Telegram) هنوز با محیط واقعی تست نشده** — همان‌طور که در
   مستندات اولیه‌اش هم ذکر شده بود.

## این‌ها را در سرویس‌های دیگر هم نوشتم
- `apimanager/NEEDS.md` — نیاز به صفحه‌ی وب برای مرور لاگ‌ها.
- `source-service/NEEDS.md` — اگر این سرویس روزی NATS گرفت (که برای قرارداد source-worker هم لازم
  می‌شود)، وصل‌کردنش به `log-collector` هم رایگان/تقریباً رایگان خواهد بود.
