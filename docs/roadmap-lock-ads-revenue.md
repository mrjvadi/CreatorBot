# نقشه‌ی راه: زنجیره‌ی قفل‌کانال / تبلیغات / درآمد

این سند بر اساس بررسی واقعی کد نوشته شده (نه فرضی). هر فاز روی فاز قبل بنا می‌شود؛
ترتیب را عوض نکن چون فاز ۳ بدون فاز ۲ معنی ندارد و فاز ۲ بدون فاز ۱ کار نمی‌کند.

## وضعیت فعلی (خلاصه‌ی بررسی)

| تکه | وضعیت |
|---|---|
| `BotInstance` ↔ کدام پلن/رایگان است | ❌ وصل نیست — `BotInstance` فقط به `TemplateID` (نوع ربات) وصل است، نه به پلن |
| bot فرعی بداند خودش رایگان است یا نه | ❌ `engine` هیچ‌جا این را از DB نمی‌خواند |
| member-bot (قفل + check-bot + cache عضویت) | ✅ کامل نوشته شده، ولی به‌صورت ربات **کاربرپسند مستقل** — owner مستقیم با آن کار می‌کند |
| ads-bot (کمپین CPJ، fake-detection، کمیسیون ۲۰٪) | ✅ کامل نوشته شده، مستقل از member-bot |
| community-service (گروه + InviteLink attribution + تقسیم ۵۰/۴۰/۱۰) | ✅ کامل، به ads-bot و revenue-service وصل با NATS |
| revenue-service (تقسیم نهایی به botpay) | ✅ کامل |
| fraud-engine (امتیاز کیفیت گروه/کاربر) | ✅ کامل |
| «ربات رایگان با قفل اختصاصی که ads-bot مدیریت کند» | ❌ مفهوم اصلاً وجود ندارد |
| تأیید ادمین روی تبلیغ ربات رایگان + تحویل به خریدار | ❌ وجود ندارد |
| ارتباط ads-bot ↔ member-bot | ❌ صفر — دو سیستم کاملاً ایزوله |

نتیجه: زیرساخت مالی/attribution (community-service, revenue-service, fraud-engine) قوی و
آماده است. چیزی که باید ساخته شود، **لایه‌ی هماهنگ‌کننده‌ی بالای آن** است — دقیقاً مدلی
که توضیح دادی (قفل رایگان در مقابل اجاره‌ای، تأیید ادمین، تحویل ربات).

---

## فاز ۱ — پایه: instance بداند به چه چیزی وصل است
**چرا اول این:** بدون این، هیچ bot فرعی نمی‌تواند تشخیص دهد قفلش رایگان است یا اجاره‌ای.

- افزودن `PlanID *uuid.UUID` و `LockMode` (free | rented | none) به `BotInstance`
- engine موقع `Start()` این را از DB می‌خواند و در `Config` قرار می‌دهد
- خروجی قابل‌تست: یک bot فرعی لاگ بزند «این instance: free» یا «rented»

## فاز ۲ — اتصال قفل به ads-bot (نه member-bot مستقیم)
**چرا قبل از فاز ۳:** باید اول بدانیم «کدام کانال» قفل است، قبل از اینکه بپرسیم «کاربر عضو است یا نه».

- در ads-bot: مدل `FreeBotSlot` (یا مشابه) — کدام instance های رایگان به کدام کمپین/کانال وصل‌اند
- جریان اجاره: کسی در ads-bot یک "rent lock" می‌خرد → کمپین تأیید ادمین می‌خواهد (مدل `Campaign.ReviewedAt` از قبل هست، استفاده می‌شود)
- بعد از تأیید: لیست instance های رایگان به آن کمپین "متصل" می‌شوند (فیلد `LockMode=rented` در فاز ۱ آپدیت می‌شود)

## فاز ۳ — member-bot از حالت کاربرپسند به زیرساخت داخلی
**چرا بعد از فاز ۲:** الان معلوم است کدام کانال‌ها قفل‌اند؛ حالا باید چک عضویت متمرکز شود.

- حذف/غیرفعال‌کردن منوهای owner در member-bot (`/newlock`, `/mybots` و...)
- یک NATS responder جدید: `member.check` — bot فرعی می‌پرسد «uid X عضو کانال Y هست؟»
- همان `MemberVerification` cache فعلی استفاده می‌شود (از قبل هست، فقط مسیر ورودی عوض می‌شود)
- bot های فرعی (uploader و...) به‌جای چک مستقیم تلگرام، از `member.check` می‌پرسند

## فاز ۴ — attribution گروه‌ها (همان InviteLink که از قبل هست)
**چرا اینجا:** فاز ۳ باید کار کند تا بدانیم کاربر واقعاً عضو شد؛ این فاز فقط می‌گوید *از کجا* آمد.

- هر گروهی که در یک کمپین شرکت می‌کند، یک `InviteLink` از community-service می‌گیرد (مکانیزم از قبل هست)
- وقتی کاربر از طریق آن لینک وارد کانال هدف شد → `membership.joined` با `campaign_id` واقعی (نه هاردکد `"organic"`) publish شود
- این یعنی فیکس یک باگ موجود: `member-bot/internal/events/publisher.go` همیشه `source:"organic"` می‌فرستد؛ باید واقعی شود

## فاز ۵ — جریان پول کامل (پاداش به کاربر + کسر از خریدار)
**چرا آخر:** بدون فاز ۱-۴، این فاز چیزی برای محاسبه ندارد.

- اتصال `membership.joined` (با campaign_id واقعی از فاز ۴) → `campaign.revenue.generated` (ads-bot از قبل publish می‌کند)
- community-service.DistributeRevenue از قبل کار می‌کند (۵۰/۴۰/۱۰) — فقط باید ورودی واقعی بگیرد
- پاداش به کاربر عضوشونده: از طریق natspayclient (که قبلاً ساختیم) با metadata شفاف کسر/واریز شود

## چرا این ترتیب و نه ترتیب دیگر
هر فاز "ورودی" فاز بعدی را می‌سازد:
فاز۱ (instance می‌داند چیست) → فاز۲ (کدام کانال قفل است) → فاز۳ (کاربر واقعاً عضو شد؟) →
فاز۴ (از کدام گروه آمد؟) → فاز۵ (چقدر پول جابه‌جا شود؟).
شروع از فاز ۳ یا ۵ بدون قبلی‌ها یعنی کد به چیزی وصل می‌شود که هنوز وجود ندارد.
