# fraud-engine

## این سرویس چیست
موتور امتیازدهی کیفیت/اعتماد — برای هر کاربر (`UserProfile`) و هر گروه/کانال (`CommunityScoreSnapshot`) یک امتیاز نگه می‌دارد که سایر سرویس‌ها (خصوصاً `ads-bot`) برای تشخیص تقلب و تعیین ضریب اعتبار پاداش از آن استفاده می‌کنند.

## مسئولیت‌ها
- ساخت/به‌روزرسانی پروفایل کاربر بر اساس رویدادهای عضویت/فعالیت (`user_profiles`, `user_profile_history`, `user_activity` در MongoDB).
- محاسبه‌ی `TrustScore` (۰ تا ۱۰۰) و برچسب (`high_risk`/`suspicious`/`normal`/`trusted`) با `internal/scorer/user_scorer.go`.
- امتیاز کیفیت گروه/کانال با `internal/scorer/community_scorer.go`.
- ثبت `FraudEvent` وقتی الگوی مشکوک تشخیص داده شود؛ این‌ها روی NATS (`fraud.detected`) منتشر می‌شوند تا `ads-bot` پاداش معلق را لغو کند.
- API عمومی برای خواندن امتیاز (بدون auth — عمداً، چون سرویس‌های دیگر باید بتوانند سریع بخوانند) و API ادمین (پشت `adminKey`) برای دیدن جزئیات/recalculate.

## ارتباطات
- HTTP: `GET /score/user/:id`, `GET /score/community/:id` (عمومی)؛ `/admin/*` پشت `authMiddleware`.
- NATS: subscribe به رویدادهای عضویت/فعالیت جامعه؛ publish `fraud.detected` (مصرف‌کننده: `ads-bot`).
- MongoDB برای پروفایل/تاریخچه.

## ایرادها و نکات
- **باقی‌مانده، کم‌خطر (مستند در گزارش امنیتی ۲۰۲۶-۰۷-۰۲)**: رویدادهای ورودی از NATS (`community.activity.updated`, `fraud.event.*`) هیچ احراز هویت سرویس ندارند — کسی با دسترسی به NATS نظری می‌تواند رویداد فعالیت جعلی بفرستد و امتیاز اعتماد یک کاربر را قبل از تسویه‌ی پاداش در `ads-bot` مصنوعی بالا ببرد. این ریسک low severity طبقه‌بندی شد چون به یک زنجیره‌ی دیگر (تسویه‌ی مالی در ads-bot) وابسته است و بخشی از ضعف عمومی نبودِ ACL در NATS است، نه چیزی خاص این سرویس. مسیر مستقیم پرداخت این‌جا وجود ندارد.
- endpoint های عمومی (`/score/*`) عمداً بدون auth هستند تا خواندن سریع باشد — این یک تصمیم طراحی است، نه یک نشتی؛ فقط داده‌ی امتیاز (نه داده‌ی شخصی حساس) برمی‌گردانند.
