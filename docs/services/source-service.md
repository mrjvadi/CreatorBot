# source-service

## این سرویس چیست
سرویسی ناتمام (stub) برای فوروارد خودکار فایل از یک کانال منبع به یک کانال تحویل، با استفاده از MTProto (کتابخانه‌ی `gotd/td`) به‌جای Bot API — یعنی با یک اکانت کاربری واقعی تلگرام («یوزربات»)، نه یک ربات. این تنها راهی است که می‌شود به پیام‌های کانال‌هایی که ربات در آن‌ها ادمین نیست هم دسترسی داشت.

## مسئولیت‌ها (طبق طراحی، هنوز پیاده‌سازی نشده)
- `internal/userbot/userbot.go` — قرار است با `gotd/td` لاگین کند و پیام‌های جدید کانال منبع را بشنود و به کانال تحویل فوروارد کند؛ `Start()` و `ForwardToDelivery()` فعلاً فقط لاگ «TODO: implement gotd/td» می‌زنند و کاری انجام نمی‌دهند.
- `internal/api/server.go` — یک HTTP API با ۳ روت (`POST /files`, `GET /files/:uuid`, `POST /files/:uuid/cache`) پشت یک `X-API-Key` ساده؛ همه‌ی handler ها فعلاً مقدار ثابت `"TODO"` برمی‌گردانند.
- مدل‌های `ArchiveFile`/`BotFileCache` تعریف شده‌اند ولی چیزی هنوز در آن‌ها نمی‌نویسد.

## ارتباطات
- HTTP روی یک پورت پیکربندی‌شده، پشت `X-API-Key`.
- PostgreSQL و Redis وصل می‌شوند ولی عملاً بلااستفاده‌اند تا وقتی منطق واقعی پیاده شود.

## ایرادها و نکات
- این سرویس دقیقاً همان‌طور که در CLAUDE.md ریشه‌ی پروژه توصیف شده، یک stub واقعی است — بدون منطق کارکردی. بررسی مستقیم کد این را تأیید می‌کند.
- **نکته‌ی حقوقی/سیاست تلگرام**: خودِ کد یک هشدار صریح دارد: «⚠️ Userbot violates Telegram ToS — use only for personal archiving» (نقض قوانین تلگرام؛ فقط برای آرشیو شخصی استفاده شود). قبل از فعال‌سازی این سرویس برای مشتریان واقعی پلتفرم (نه فقط استفاده‌ی شخصی)، این ریسک باید آگاهانه بررسی و پذیرفته شود — ریسک مسدودشدن اکانت/شماره‌ی تلفن استفاده‌شده برای MTProto واقعی است.
- لایه‌ی auth ساده‌ی HTTP آن (`X-API-Key`) درست پیاده شده — مشکل امنیتی خاصی در همین اسکلت فعلی دیده نشد؛ ریسک اصلی صرفاً «ناتمام‌بودن» است، نه یک آسیب‌پذیری فعال.
