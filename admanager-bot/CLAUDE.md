# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`admanager-bot` is a single-owner Telegram bot for scheduling and posting ads across a set of Telegram channels that the owner administers. It is **not** a marketplace: there is no advertiser/customer role, no wallet, no budget, and no approval workflow. A "campaign" here just means a rotation schedule (interval, rotation, targeting by tag/channel) applied to a set of ad posts, all owned by the single admin identified by `OWNER_ID`. Every text/callback handler is gated by `isAdmin` (`c.Sender().ID == h.ownerID`) — there is exactly one authorized user.

This is one bot among several sibling bots (e.g. `ads-bot`) that share `../shared` and `../shared-core` Go modules but otherwise do not share code, database, or business logic with them. Do not assume concepts from other bots (multi-publisher wallets, payment campaigns, etc.) apply here — see the comment block at the top of `internal/models/models.go` and the analysis in `NEEDS.md` for the explicit contrast.

Read `PLAN.md` for the original phase-by-phase implementation plan (phases 1–8 are done) and `NEEDS.md` for a dated, file:line-referenced audit of what's incomplete or dead code as of 2026-07-04 (morning) — note that `FEATURES_V2.md` (also 2026-07-04, later the same day) is a second round of requirements gathered directly from the owner and its phases A–H have since been implemented on top of what `NEEDS.md` describes, so treat `FEATURES_V2.md` as the most current design reference for the reply-cycle/scheduling/ad-editing behavior, and `NEEDS.md` as current for everything it doesn't touch (stats wiring, tests, `TotalClicks`).

## Build & run

This module lives inside a larger workspace and depends on sibling modules via `replace` directives in `go.mod` (`../shared`, `../shared-core`), so it must be built from a checkout where those two directories exist alongside `admanager-bot/`.

```bash
go build ./...               # build everything
go vet ./...                 # static checks
go test ./internal/models/... # only package with tests today (pure logic, no external deps)
go run ./cmd/bot              # run the bot (needs .env — copy from .env.example)
gofmt -l .                    # formatting check
```

`internal/models/schedule_test.go` covers the pure daily-window/rotation-index math (`InDailyWindow`, `CurrentWindowStart`, `DailyWindowBounds`, `RotationIndex` — all in `internal/models/schedule.go`), and `internal/models/parse_test.go` covers the campaign schedule text-parsing (`ParseSchedule`, `ParseClockRange`, `ParseClock`, `NormalizeDigits` — in `internal/models/parse.go`). Both files are dependency-free (only stdlib `time`/`strconv`/`strings`), so `go test ./internal/models/...` runs without touching Mongo/Telegram. `internal/scheduler`, `internal/tgbot/handler_schedule.go`, and `internal/tgbot/handler_campaign.go` all call into these same functions rather than duplicating the logic — if you need to change schedule-window or rotation behavior, change it once in `internal/models` and everything downstream picks it up. Nothing else in the repo has tests yet — flagged in `NEEDS.md` as a remaining gap, particularly around the scheduler's job-execution side (`sendAd`/`sendReply`/`endCycle`) which still isn't covered. There is no Makefile/README; the commands above are the plain `go` toolchain.

Configuration is env-based (`.env`, loaded via `github.com/mrjvadi/creatorbot/shared/pkg/config`). Required vars are documented in `.env.example`: `BOT_TOKEN`, `OWNER_ID` (the only Telegram user ID allowed to use the bot), plus Postgres/Mongo/Redis/NATS connection info consumed by the shared `engine` package.

## Architecture

**Layering**: `cmd/bot/main.go` wires everything — builds a `shared-core/engine.Engine` (Mongo, Redis cache, Postgres heartbeat, NATS, logger), constructs the telebot instance, then wires `tgbot.NewHandler` (Telegram handlers) and `scheduler.New` (background job runner) on top of the same `store.Store`. `internal/store` is the only layer that talks to MongoDB.

**Data isolation**: every Mongo document has an `instance_id` field, and every query goes through `Store.f(...)` (in `internal/store/store.go`), which injects `instance_id` into the filter automatically. This is how multiple bot instances/deployments share one Mongo cluster without seeing each other's data. When adding a new store method, always build filters through `s.f(...)`, never a bare `bson.D`.

**Domain model** (`internal/models/models.go`): `Channel`/`Tag` (channels the admin manages, taggable for targeting), `Campaign` (schedule + targeting, no money — `StartHour`/`StartMinute`/`EndHour`/`EndMinute` define a daily send window that can wrap past midnight, see `EndAt` for an optional absolute campaign-wide end), `Advertisement` (one main message + `Replies []AdReply`, each reply carrying its own `DurationMinutes`, stored as `source_chat_id` + message IDs from the admin's own chat with the bot — content is reproduced via `bot.Copy`, never Telegram's native forward, so no "forwarded from" label appears; also carries `KeepAsLastMessage`/`PinMessage`/`DeletePreviousOnRepost` for the "fixed ad" behavior), `ScheduledJob` (generic queued work: send_ad/send_reply/delete_post/update_stats/end_campaign/reminder), `Reservation` (one send slot: campaign × channel × ad; `LiveMessageIDs`/`CurrentReplyIndex` track what's *currently* visible in the channel right now, while `PostedMessageIDs` is an append-only history of everything ever posted for that slot), `CampaignTemplate` (canned campaign defaults, including the daily window), `BotSettings` (per-instance singleton config, including `ReminderMinutesBefore`), `AuditLog`.

**Telegram handler layer** (`internal/tgbot/`): `bot.go` defines `Handler`/`Deps` and registers `/start`, `/help`, free text, media, and callback handlers. `router.go`'s `onCallback` is a single big switch on a `action:arg` callback-data convention (`\f`-prefixed by telebot, split on first `:`) — new inline buttons should follow this `prefix_verb:id` naming (e.g. `ch_toggle`, `cmp_pause`, `ad_del`). `state.go` implements a per-user wizard state machine persisted in Redis (`admanager:state:<instance>:<uid>`, 15 min TTL, JSON) for multi-step flows (new campaign, new ad, add channel, etc.); `handleStep` dispatches on the current `step`. There's also a separate small Redis-backed "ctx" store (`setCtx`/`getCtx`) used to remember a parent ID (campaign/channel/ad) across toggle callbacks, because Telegram callback_data is capped at 64 bytes and can't hold two UUIDs. `handler_schedule.go` renders a day-by-day timeline of projected send slots (computed analytically from each running campaign's schedule, not read from `Reservation` documents, since reservations only exist a few minutes ahead of the actual send — see `FEATURES_V2.md` §2.7 for why). `handler_settings.go` is the one real settings screen so far (the reminder lead time); everything else in `BotSettings` (welcome/help message overrides) is still only editable by writing to Mongo directly.

**Scheduler** (`internal/scheduler/scheduler.go`): a single goroutine ticks every 30s and does two things each tick: (1) fetch due `ScheduledJob`s and run them concurrently (`go s.runJob(job)`), marking them `running` first for idempotency across restarts; (2) `planCampaigns` scans all `running` campaigns. For each campaign it first checks `inDailyWindow` (handles midnight wraparound); if the window just closed it calls `cutOffCampaign` (deletes any still-live messages, cancels pending/confirmed reservations and all pending jobs for that campaign) and moves on — this only affects that one campaign, not the bot or other campaigns. If the window is open, it resolves target channels (by tag and/or explicit channel IDs, respecting `MinMemberCount`), picks the "current" ad via a time-based rotation index, and computes each channel's `nextDue` slot (`lastReservation.ScheduledAt + cycleGap`, or the window start if there's no prior reservation). If `BotSettings.ReminderMinutesBefore > 0`, the `Reservation` + `send_ad` job are created that many minutes *ahead* of `nextDue` (not at `nextDue` itself), plus a `JobTypeReminder` job that DMs the owner shortly before the real send; with reminders off, it falls back to creating them right at `nextDue`, matching the old immediate-send behavior. Sending the main post schedules a `delete_post` "ceiling" job at `DeleteAfterMinutes` from now (whole-cycle safety net) and, if there are replies, the first reply at `IntervalMinutes` out. Each `sendReply` deletes whatever reply was previously live before posting the next one (only ever one reply visible at a time), then schedules either the next reply (after its own `DurationMinutes`) or a final `delete_post` if it was the last one. `endCycle` (shared by the ceiling job and the natural last-reply path) deletes whatever's currently live, marks the reservation `expired`, and cancels any other pending job for that same reservation so the two paths can't double-fire. `afterPost` runs after every successful send and re-posts any other live reservation in that channel belonging to an ad with `KeepAsLastMessage` set, so it stays the newest message (with `PinMessage`/`DeletePreviousOnRepost` applied); this is deliberately not recursive (to avoid two fixed ads ping-ponging forever) — see `FEATURES_V2.md` §2.4/§2.7 for the known limitation around ordering multiple simultaneous fixed ads. Failed jobs retry with backoff (`(Attempts+1) * 5 min`) up to `MaxAttempts`, then mark both the job and its associated reservation failed.

**Known incomplete/dead paths** (see `NEEDS.md` for exact file:line — most of these predate `FEATURES_V2.md` and are still true): `update_stats` jobs are never actually scheduled by any code path (only the `case` and the standalone function exist); `AvgViews`/`EngageRate` are read but never written; `Campaign.TotalClicks` is defined but never read or written (only impressions are tracked/shown). `JobTypeReminder` is **no longer dead** — it's now scheduled from `planCampaigns` and handled by `Scheduler.sendReminder`. The settings menu is **no longer a pure stub** — `⚙️ تنظیمات` now has one real, working control (reminder lead time via `handler_settings.go`); everything else in `BotSettings` is still not editable from the menu. Don't assume any of this is wired up just because the type/model exists — grep first.
